The Zero-Frequency Problem in Text Compression

import math
from collections import Counter, defaultdict

1. Training Corpus

We intentionally use a tiny corpus to force sparsity.

training_text = """
the cat sat on the mat
the cat sat on the mat
the cat sat on the mat
the cat sat on the
""".strip().split()

counts = Counter(training_text)
N = sum(counts.values())
counts, N
(Counter({'the': 8, 'cat': 4, 'sat': 4, 'on': 4, 'mat': 3}), 23)

2. Unigram MLE and Compression Failure

Maximum Likelihood Estimation (MLE) estimates probabilities by counting:

\[P(w) = \frac{\text{count}(w)}{\text{total words}}\]

This is intuitive: if ‚Äúthe‚Äù appears 10 times out of 25 words, we estimate \(P(\text{the}) = 10/25 = 0.4\).

The problem: What if a word never appeared in training? Its count is 0, so:

\[P(\text{rug}) = \frac{0}{25} = 0\]

A probability of zero seems reasonable (‚Äúwe never saw it!‚Äù), but it creates a catastrophic problem for compression‚Ä¶

def mle_prob(word, counts, N):
    return counts[word] / N

novel_word = "rug"
print("MLE P(rug) =", mle_prob(novel_word, counts, N))
MLE P(rug) = 0.0

A Quick Detour: Why is this Catastrophic for Compression?

Before we see why zero probability breaks everything, we need to understand what we mean by ‚Äúbits required to encode‚Äù something.

Shannon‚Äôs Key Insight (1948): Information is surprise.

  • If I tell you ‚Äúthe sun rose today‚Äù ‚Äî no surprise, no information. You already knew that.
  • If I tell you ‚Äúa meteor struck campus‚Äù ‚Äî huge surprise, lots of information.

Bits measure surprise mathematically:

\[\text{bits}(w) = -\log_2 P(w)\]

Event Probability Bits Intuition
Coin lands heads 0.5 1 bit One yes/no question: ‚ÄúHeads?‚Äù
Roll a 6 on a die 1/6 ‚âà 0.167 ~2.6 bits Between 2-3 yes/no questions
Specific card from deck 1/52 ‚âà 0.019 ~5.7 bits About 6 yes/no questions
Win the lottery 0.0000001 ~23 bits Enormous surprise!

Why this matters for compression:

Compression algorithms like arithmetic coding or Huffman coding work by assigning shorter codes to common symbols and longer codes to rare ones. The formula \(-\log_2 P(w)\) is the theoretical minimum ‚Äî you literally cannot do better.

  • Common words like ‚Äúthe‚Äù (high probability) ‚Üí few bits ‚Üí short codes
  • Rare words like ‚Äúquixotic‚Äù (low probability) ‚Üí more bits ‚Üí longer codes
  • Unseen words (zero probability) ‚Üí infinite bits ‚Üí impossible to encode

This is why probability models matter for compression: your probability estimates directly determine your code lengths.

The Catastrophe: Zero Probability

Now we see the problem: when \(P = 0\), we get \(-\log_2(0) = \infty\) bits.

The encoder literally cannot proceed. Arithmetic coding works by subdividing the number line according to symbol probabilities. A zero-probability symbol gets zero space ‚Äî there‚Äôs nowhere to put it. The algorithm crashes.

def code_length(prob):
    return -math.log2(prob)

try:
    code_length(mle_prob(novel_word, counts, N))
except ValueError as e:
    print("Compression failure:", e)
Compression failure: math domain error

‚òùÔ∏è This is the zero-frequency problem in action. Python raises a ValueError because math.log2(0) is undefined.

The fix? We need to ensure every word has non-zero probability, even words we‚Äôve never seen. This is called smoothing.

3. Laplace Smoothing (Unigrams)

Laplace smoothing (also called ‚Äúadd-one smoothing‚Äù) is the simplest fix: pretend every word was seen at least once.

\[P_{\text{Laplace}}(w) = \frac{\text{count}(w) + 1}{N + V}\]

Where: - +1 in numerator: Every word gets one ‚Äúfree‚Äù count - +V in denominator: We add \(V\) (vocabulary size) to keep probabilities summing to 1

The tradeoff: We ‚Äústeal‚Äù probability mass from frequent words and redistribute it to rare/unseen words. Frequent words get slightly lower probabilities than MLE would give them.

V = len(counts)

def laplace_prob(word, counts, N, V):
    return (counts[word] + 1) / (N + V)

for w in ["the", "cat", "sat", "mat", "rug"]:
    print(f"{w:>4}: {code_length(laplace_prob(w, counts, N, V)):.2f} bits")
 the: 1.64 bits
 cat: 2.49 bits
 sat: 2.49 bits
 mat: 2.81 bits
 rug: 4.81 bits

‚òùÔ∏è Now ‚Äúrug‚Äù has a finite code length! It takes more bits than common words (as it should), but it‚Äôs no longer impossible to encode.

Notice the cost: ‚Äúthe‚Äù now takes slightly more bits than pure MLE would suggest, because we redistributed some of its probability to unseen words.

4. Move to Bigrams

Now the sparsity problem becomes much worse.

Why Bigrams Make Everything Worse

With unigrams, we have \(V\) possible words. With bigrams, we have \(V^2\) possible pairs!

Model Possible Events Our Vocab (V=6)
Unigram \(V\) 6 words
Bigram \(V^2\) 36 pairs
Trigram \(V^3\) 216 triples

Most of those \(V^2\) pairs will never appear in training data. The zero-frequency problem explodes combinatorially.

bigrams = list(zip(training_text[:-1], training_text[1:]))
bigram_counts = Counter(bigrams)
unigram_counts = Counter(training_text)

bigram_counts
Counter({('the', 'cat'): 4,
         ('cat', 'sat'): 4,
         ('sat', 'on'): 4,
         ('on', 'the'): 4,
         ('the', 'mat'): 3,
         ('mat', 'the'): 3})

‚òùÔ∏è Look at the bigram counts. We only see a handful of the 36 possible pairs. Most combinations like (‚Äúmat‚Äù, ‚Äúcat‚Äù) or (‚Äúon‚Äù, ‚Äúsat‚Äù) have count zero.

5. Bigram MLE

Probability of w2 given w1.

Bigram MLE uses conditional probability ‚Äî the probability of word \(w_2\) given that we just saw \(w_1\):

\[P(w_2 | w_1) = \frac{\text{count}(w_1, w_2)}{\text{count}(w_1)}\]

We divide by the count of the context word, not the total corpus size. This asks: ‚ÄúOf all the times we saw \(w_1\), how often was it followed by \(w_2\)?‚Äù

def bigram_mle(w1, w2, bigram_counts, unigram_counts):
    return bigram_counts[(w1, w2)] / unigram_counts[w1]

print("P(cat | the) =", bigram_mle("the", "cat", bigram_counts, unigram_counts))
print("P(rug | the) =", bigram_mle("the", "rug", bigram_counts, unigram_counts))
P(cat | the) = 0.5
P(rug | the) = 0.0

‚òùÔ∏è Key insight: Even though ‚Äúthe‚Äù is common and ‚Äúrug‚Äù could plausibly follow it, we get \(P(\text{rug}|\text{the}) = 0\) because that specific combination never appeared in training.

This is worse than the unigram case ‚Äî we don‚Äôt need a novel word to hit zero probability, just a novel sequence of known words!

6. Bigram Compression Failure

try:
    code_length(bigram_mle("the", "rug", bigram_counts, unigram_counts))
except ValueError as e:
    print("Bigram compression failure:", e)
Bigram compression failure: math domain error

‚òùÔ∏è Same crash, different cause. This time it‚Äôs not a novel word ‚Äî both ‚Äúthe‚Äù and ‚Äúrug‚Äù are in our vocabulary. It‚Äôs a novel bigram.

7. Laplace Smoothing for Bigrams

Bigram Laplace smoothing applies the same +1 trick, but now we add \(V\) to account for all possible next words:

\[P_{\text{Laplace}}(w_2 | w_1) = \frac{\text{count}(w_1, w_2) + 1}{\text{count}(w_1) + V}\]

Why +V? After seeing \(w_1\), there are \(V\) possible words that could follow. We‚Äôre adding 1 imaginary count for each possibility.

def bigram_laplace(w1, w2, bigram_counts, unigram_counts, V):
    return (bigram_counts[(w1, w2)] + 1) / (unigram_counts[w1] + V)

for w2 in ["cat", "rug"]:
    p = bigram_laplace("the", w2, bigram_counts, unigram_counts, V)
    print(f"P({w2} | the) = {p:.4f} ‚Üí {code_length(p):.2f} bits")
P(cat | the) = 0.3846 ‚Üí 1.38 bits
P(rug | the) = 0.0769 ‚Üí 3.70 bits

‚òùÔ∏è Both bigrams now have finite code lengths. ‚Äúrug‚Äù following ‚Äúthe‚Äù costs more bits (it‚Äôs unexpected), but it‚Äôs encodable.

But there‚Äôs a problem‚Ä¶ Laplace smoothing is quite aggressive. Adding 1 to every possible bigram means we‚Äôre giving a lot of probability mass to things that will probably never happen. This becomes wasteful with larger vocabularies.

8. Why Good‚ÄìTuring and Backoff Exist

Laplace smoothing has problems at scale. Real systems use smarter techniques:

Backoff: If we haven‚Äôt seen a bigram, fall back to the unigram probability. Only use the lower-order model when the higher-order one has no data.

\[P(w_i | w_{i-1}) = \begin{cases} P_{\text{bigram}}(w_i | w_{i-1}) & \text{if count}(w_{i-1}, w_i) > 0 \\ \alpha \cdot P_{\text{unigram}}(w_i) & \text{otherwise} \end{cases}\]

Interpolation: Instead of choosing one or the other, blend all n-gram levels together:

\[P(w_i | w_{i-1}) = \lambda_1 \cdot P_{\text{bigram}}(w_i | w_{i-1}) + \lambda_2 \cdot P_{\text{unigram}}(w_i)\]

where \(\lambda_1 + \lambda_2 = 1\). This way, even seen bigrams get some influence from unigram statistics.

Discounting: Instead of adding counts (like Laplace), subtract a small amount from observed counts and redistribute that probability mass to unseen events. Good‚ÄìTuring estimates how much to discount based on how many things occurred exactly once, twice, etc.

\[P_{\text{discounted}}(w) = \frac{\text{count}(w) - d}{N}\]

(Jurafsky & Martin chapter 3 covers these in detail.)

The Problem with Laplace Smoothing at Scale

Imagine a vocabulary of \(V = 50{,}000\) words: - Possible bigrams: \(V^2 = 2.5\) billion - Possible trigrams: \(V^3 = 125\) trillion

Laplace smoothing adds 1 to all of these ‚Äî even absurd combinations like ‚Äúthe the the‚Äù or ‚Äúbanana quantum fishbowl.‚Äù We‚Äôre wasting probability mass on events that will never occur.

But there‚Äôs a subtler problem: what probability should we use when backing off?

The ‚ÄúSan Francisco‚Äù Problem: Why Context Matters for Backoff

Consider these two words in a large corpus:

Word Raw Count Contexts It Follows
Francisco 5,000 ‚ÄúSan‚Äù (4,990), ‚ÄúPope‚Äù (10)
Tuesday 5,000 ‚Äúlast‚Äù, ‚Äúnext‚Äù, ‚Äúevery‚Äù, ‚Äúon‚Äù, ‚Äúuntil‚Äù, ‚Ä¶ (hundreds)

Both words appear 5,000 times. But look at their contexts: - ‚ÄúFrancisco‚Äù almost only follows ‚ÄúSan‚Äù ‚Äî it‚Äôs trapped in one bigram - ‚ÄúTuesday‚Äù follows hundreds of different words ‚Äî it‚Äôs versatile

Now imagine we need to back off:

We encounter a novel bigram like ‚ÄúI visited Francisco‚Äù ‚Äî we‚Äôve never seen ‚Äúvisited Francisco‚Äù in training. We need to estimate P(Francisco | visited) by backing off to some unigram-like probability.

The problem with raw frequency: If we use raw counts, P(Francisco) ‚âà P(Tuesday). But intuitively, ‚ÄúI visited Tuesday‚Äù makes more sense than ‚ÄúI visited Francisco‚Äù (without ‚ÄúSan‚Äù).

Kneser-Ney‚Äôs insight: Instead of raw frequency, count how many different contexts a word appears in (its continuation count):

\[P_{\text{continuation}}(w) = \frac{|\{v : \text{count}(v, w) > 0\}|}{|\text{all bigram types}|}\]

  • ‚ÄúFrancisco‚Äù: appears after ~2 distinct words ‚Üí low continuation probability
  • ‚ÄúTuesday‚Äù: appears after ~500 distinct words ‚Üí high continuation probability

This is why Kneser-Ney outperforms Laplace: when backing off, it asks ‚Äúhow versatile is this word?‚Äù not ‚Äúhow frequent is this word?‚Äù

9. LLM Connection

  • Subword tokenization limits true novelty
  • Parameter sharing smooths across contexts
  • Softmax ensures no zero probabilities

Same zero-frequency problem ‚Äî different machinery.

From N-grams to Neural Language Models

Modern LLMs like GPT and Claude face the exact same zero-frequency problem ‚Äî but solve it differently:

Classical N-grams Modern LLMs
Discrete word counts Continuous embeddings
Explicit smoothing (+1, backoff) Implicit smoothing via parameter sharing
Fixed context (n-1 words) Attention over long contexts
Zero prob for unseen n-grams Softmax ensures all tokens > 0

The fundamental insight remains: you can‚Äôt assign zero probability to anything you might need to encode.

üéØ Key Takeaways

  1. Zero counts ‚Üí zero probability ‚Üí infinite bits ‚Äî compression is impossible
  2. Smoothing redistributes probability mass to ensure nothing has \(P = 0\)
  3. Higher-order n-grams make sparsity exponentially worse (\(V^n\) possible events)
  4. Laplace smoothing is simple but wasteful; real systems use smarter methods
  5. LLMs solve the same problem with neural networks, but the core issue is identical
